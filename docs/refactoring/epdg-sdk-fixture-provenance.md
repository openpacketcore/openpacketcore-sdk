# ePDG SDK Fixture Provenance Map

This document classifies the test bytes, fixtures, and parity corpora observed
in the ePDG import packet. It prevents later SDK tasks from treating local
product fixtures as protocol conformance evidence without the independent
provenance required by [ADR 0015](../adr/0015-protocol-codec-conformance-policy.md).

## Import roots and evidence rule

| Role | Import path | Evidence use |
| --- | --- | --- |
| Task packet root | `/tmp/openpacketcore-epdg-sdk-task-packet` | Source of task packets, shared context, and source map for this import. |
| Mirrored source root | `/tmp/openpacketcore-epdg-sdk-task-packet/epdg-source` | Read-only source for parity comparison and fixture triage. |

ADR 0015 distinguishes conformance evidence from regression/parity evidence:

- **Acceptable conformance evidence:** bytes hand-authored from a specification
  with octet-level comments and citations, or bytes captured from an
  independent implementation with source, license, redaction, and capture
  metadata.
- **Parity evidence:** bytes produced by the ePDG implementation, constructed by
  local test builders, or round-tripped through the same local codec. These are
  useful for migration compatibility but do not prove SDK wire correctness.
- **Unknown / blocked evidence:** referenced files or corpora whose origin,
  license, or packet contents are missing.

No code behavior changed as part of this map.

## Local corpus that must not be used as conformance evidence yet

- GTPv2-C packets assembled through `build_raw_packet` or local typed builders.
- GTPv2-C same-codec decode/encode round trips, even when byte-exact.
- The `roundtrip_messages.rs` realistic S5/S8 capture until capture origin,
  peer implementation, license, and redaction notes are present.
- Diameter CER/CEA, DWR/DWA, DPR/DPA, Rf, Gx, and S6a messages generated by
  local builder APIs.
- IKEv2 messages built by `IkeMessageBuilder` or payload constructors in local
  tests.
- The referenced StrongSwan IKE_SA_INIT fixture until the missing file is
  restored and documented.
- ePDG smoke/lab UE packets from `eg-epdg-testkit` unless a later task records
  packet bytes, source, redaction status, and whether the peer is independent.
- Product conformance pages under `docs/conformance`; they remain product claim
  context, not SDK proof.

## Fixture intake checklist for follow-up protocol tasks

Every future `opc-proto-*` fixture should carry:

1. Protocol, message name, and release or RFC section.
2. Provenance class: `spec-authored`, `independent-capture`, `epdg-parity`,
   `synthetic-negative`, or `unknown`.
3. Source citation or capture metadata, including independent implementation and
   version when applicable.
4. License/permission statement for captured bytes.
5. Redaction statement confirming no real subscriber identifiers, key material,
   Diameter Session-Id values, LI IDs, or deployment secrets remain.
6. Byte-level comments for conformance fixtures.
7. Expected decode result and expected re-encode behavior, including any declared
   canonicalization.
8. Raw/unknown element preservation expectation.
9. Hostile-input expectation for truncation, overflow, duplicate, recursion, or
   count-limit cases.
10. Fuzz seed inclusion decision and corpus digest where used.

## Residual provenance gaps

- The packet mirror did not include `common/crates/protocol/eg-ikev2-parser/tests/fixtures/strongswan_ike_sa_init.bin`, even though parser and encode tests reference it.
- Several ePDG crates used as harvest sources do not declare an explicit
  `license` field in their packet `Cargo.toml`; copying code from those areas
  must wait for source-owner confirmation or a repo-level license artifact.
- No source packet in this inventory supplied an SDK-ready fixture manifest with
  ADR 0015 provenance labels, redaction status, and capture metadata. The first
  protocol fixture-port tasks must create that manifest before claiming
  conformance.
